Pourquoi le DNS d'abord ?

Active Directory repose entièrement sur le DNS (les fameux enregistrements SRV du Jour 2). Sans un DNS sain pour formation.local, la promotion échoue ou le domaine devient injoignable. Le Jour 2 n'était pas un hasard : il préparait celui-ci.

Active Directory, qu'est-ce que c'est ?

La métaphore du service RH

Active Directory Domain Services (AD DS), c'est le service des ressources humaines d'une entreprise, version informatique. Un annuaire central qui sait qui est qui (les comptes), quelles machines existent, qui appartient à quel groupe, et surtout qui a le droit de faire quoi. Au lieu de gérer chaque poste séparément, on gère tout depuis un point unique : le siège.

Concrètement, AD assure quatre choses : l'authentification (vérifier votre badge à l'entrée), l'autorisation (décider à quelles pièces votre badge ouvre), la centralisation (un seul compte pour toutes les machines), et la réplication (plusieurs sièges qui se synchronisent pour résister aux pannes).

La hiérarchie : forêt, domaine, OU, objet

Du groupe à la fiche employé

Reprenons l'entreprise. La forêt est le groupe tout entier : la plus grande frontière de sécurité. Un domaine est une filiale administrée de façon autonome. Un arbre regroupe des filiales partageant le même « nom de famille » (espace de noms contigu). Une unité d'organisation (OU) est un classeur de service où l'on range les gens et les machines, et dont on peut confier la clé à un responsable. Un objet est une fiche : un employé, un ordinateur, un groupe.

Le nom de la forêt = le nom de son domaine racine

Une forêt porte le nom de son premier domaine (le domaine racine). Notre forêt s'appelle donc formation.local, exactement comme le domaine que l'on va créer. On ne mélange pas un nom de forêt et un nom de domaine différents au démarrage.

Lire un nom LDAP

DC = Domain Component (un morceau du nom de domaine), OU = unité d'organisation, CN = Common Name (le nom de l'objet). On lit de droite à gauche : CN=jdupont,OU=Utilisateurs,DC=formation,DC=local = « l'objet jdupont, dans l'OU Utilisateurs, du domaine formation.local ».

Contrôleurs de domaine et rôles FSMO

Le siège et ses guichets uniques

Un contrôleur de domaine (DC) est un siège qui détient le registre AD et valide les badges. On en met plusieurs pour la disponibilité — ils se répliquent. Mais certaines opérations délicates ne peuvent être faites qu'à un seul endroit à la fois, comme certains tampons officiels qui n'existent qu'en un exemplaire : ce sont les rôles FSMO (Flexible Single Master Operations, « opérations à maître unique »).

À retenir

Sur un premier DC seul, les 5 rôles FSMO sont sur cette unique machine. On les répartit seulement quand on ajoute d'autres contrôleurs. Le catalogue global, lui, est l'index de toute la forêt : il permet de retrouver n'importe quel objet rapidement.

Exercice de conception

Avant de construire, on réfléchit à la structure. Durée : 45 min. Pour chaque scénario, proposez une structure forêt / domaine / OU, justifiez, placez les contrôleurs de domaine et définissez le nommage DNS.

  • Scénario A — PME (50 personnes, un site) : départements Direction, Comptabilité, Commercial, Technique. Une forêt, un domaine, des OU par département suffisent largement. Un DC (idéalement deux pour la disponibilité).
  • Scénario B — Groupe 3 sites (300 personnes) : autonomie par site. Un domaine unique avec des sites AD et des OU par localisation ; au moins un DC par site pour l'authentification locale et la réplication.
  • Scénario C — Holding (1000 personnes, filiales, sécurité élevée) : séparation forte des environnements (Production, Développement, DMZ). On envisage plusieurs domaines, voire plusieurs forêts pour isoler la sécurité.
Principe directeur

La forêt est la frontière de sécurité. Si deux ensembles ne doivent jamais se faire confiance, on les sépare en forêts distinctes. Tant qu'une confiance mutuelle est acceptable, un seul domaine bien découpé en OU suffit le plus souvent.

Installer le rôle AD DS

Administrator: Windows PowerShell
# Installer le rôle Active Directory Domain Services
Install-WindowsFeature -Name AD-Domain-Services -IncludeManagementTools

# Vérifier
Get-WindowsFeature -Name AD-Domain-Services
Installer n'est pas promouvoir

Installer le rôle pose seulement les outils sur le serveur. Le serveur ne devient un contrôleur de domaine qu'après l'étape de promotion (§6). Les deux gestes sont distincts.

Promouvoir le serveur en premier contrôleur

Transformer le serveur en siège

La promotion transforme un serveur ordinaire en siège qui tient le registre. Comme c'est le tout premier, il faut d'abord créer la forêt et son domaine racine. Le siège ouvre alors ses guichets : authentification, réplication, annuaire.

En interface graphique

  1. Après l'installation du rôle, le Gestionnaire de serveur affiche un drapeau d'avertissement : Promouvoir ce serveur en contrôleur de domaine.
  2. Choisir Ajouter une nouvelle forêt → nom de domaine racine : formation.local.
  3. Définir le mot de passe DSRM (mode de restauration) : P@ssw0rd123!
  4. Laisser le NetBIOS proposé : FORMATION. Valider les contrôles, puis lancer (redémarrage automatique).

En PowerShell

Administrator: Windows PowerShell
# Créer la forêt et promouvoir ce serveur en premier DC
Install-ADDSForest -DomainName "formation.local" `
                   -DomainNetbiosName "FORMATION" `
                   -InstallDns:$true `
                   -SafeModeAdministratorPassword (Read-Host -AsSecureString "Mot de passe DSRM")

Analysons cette commande ligne par ligne :

  • -DomainName "formation.local" : le nom du domaine racine (et donc de la forêt).
  • -DomainNetbiosName "FORMATION" : le nom court, à l'ancienne, utilisé pour ouvrir une session (FORMATION\jdupont).
  • -InstallDns:$true : installe et intègre le DNS au domaine (les enregistrements SRV se créent automatiquement).
  • -SafeModeAdministratorPassword : le mot de passe DSRM, demandé de façon masquée par Read-Host -AsSecureString.
Le mot de passe DSRM : la clé du coffre

Le mot de passe DSRM (Directory Services Restore Mode) est la clé de secours pour réparer AD si le domaine tombe. Il est différent du mot de passe administrateur ordinaire et doit être conservé précieusement : sans lui, pas de restauration possible.

Configuration réseau après promotion

Le siège qui s'inscrit dans son propre annuaire

Une fois promu, le serveur tient lui-même l'annuaire du domaine. Il doit donc se consulter lui-même pour résoudre les noms : son DNS primaire devient 127.0.0.1 (« moi-même »). C'est le geste que tout le monde oublie, et qui casse silencieusement le domaine.

Administrator: Windows PowerShell
# Le serveur pointe vers lui-même pour le DNS (LAN = nom de la carte interne du Jour 1)
Set-DnsClientServerAddress -InterfaceAlias "LAN" -ServerAddresses "127.0.0.1"

# Vérifier
Get-DnsClientServerAddress

# Conserver des redirecteurs pour Internet
Set-DnsServerForwarder -IPAddress "8.8.8.8","1.1.1.1"

# Tester la résolution du domaine et du DC
Resolve-DnsName formation.local
Resolve-DnsName srv-dc01.formation.local
Le piège classique du Jour 1 revisité

Au Jour 1, on laissait le DNS de la carte interne vide (Internet venait de la carte NAT). Maintenant que le serveur est le DNS du domaine, il doit pointer vers 127.0.0.1. Si vous le laissez sur un DNS externe (8.8.8.8 en primaire), il ne trouvera pas ses propres enregistrements SRV et l'ouverture de session échouera.

Étape 3 — Réseau post-promotion

Administrator: Windows PowerShell
Set-DnsClientServerAddress -InterfaceAlias "LAN" -ServerAddresses "127.0.0.1"
Set-DnsServerForwarder -IPAddress "8.8.8.8","1.1.1.1"

Étape 4 — Valider

Administrator: Windows PowerShell
# 1. Les services AD doivent tourner
Get-Service ADWS, KDC, NETLOGON | Select-Object Name, Status

# 2. Le domaine et la forêt
Get-ADDomain | Select-Object DNSRoot, NetBIOSName, DomainMode

# 3. Le contrôleur de domaine
Get-ADDomainController -Filter * | Select-Object Name, IPv4Address, OperatingSystem

# 4. Les rôles FSMO (les 5 sur cette machine)
netdom query fsmo

# 5. La résolution du domaine
Resolve-DnsName formation.local -Type A
Critères de réussite

ADWS, KDC et NETLOGON sont Running ; Get-ADDomain renvoie formation.local ; les 5 rôles FSMO pointent vers SRV-DC01. Le domaine est né.

Construire la structure d'unités d'organisation

Le classeur à intercalaires

Une OU est un intercalaire dans le grand classeur du domaine. On y range les comptes et les machines par service, ce qui permet deux choses : appliquer des règles différentes par dossier (au Jour 4 : les stratégies de groupe), et confier la clé d'un seul intercalaire à un responsable sans lui donner accès à tout le classeur (la délégation).

Administrator: Windows PowerShell
# OU racine de l'organisation
New-ADOrganizationalUnit -Name "EcoleIT" -Path "DC=formation,DC=local" `
                         -Description "OU racine École IT"

# Branche Utilisateurs et ses sous-OU
New-ADOrganizationalUnit -Name "Utilisateurs" -Path "OU=EcoleIT,DC=formation,DC=local"
New-ADOrganizationalUnit -Name "Formateurs"     -Path "OU=Utilisateurs,OU=EcoleIT,DC=formation,DC=local"
New-ADOrganizationalUnit -Name "Administration" -Path "OU=Utilisateurs,OU=EcoleIT,DC=formation,DC=local"
New-ADOrganizationalUnit -Name "Stagiaires"     -Path "OU=Utilisateurs,OU=EcoleIT,DC=formation,DC=local"

# Branche Ordinateurs
New-ADOrganizationalUnit -Name "Ordinateurs"   -Path "OU=EcoleIT,DC=formation,DC=local"
New-ADOrganizationalUnit -Name "Serveurs"      -Path "OU=Ordinateurs,OU=EcoleIT,DC=formation,DC=local"
New-ADOrganizationalUnit -Name "PostesTravail" -Path "OU=Ordinateurs,OU=EcoleIT,DC=formation,DC=local"

Comment lire ces commandes :

  • -Name = le nom de l'intercalaire à créer.
  • -Path = où on le range, en notation LDAP (lue de droite à gauche). On empile les OU= pour descendre dans l'arborescence.

Protéger contre la suppression accidentelle

Administrator: Windows PowerShell
Set-ADOrganizationalUnit -Identity "OU=EcoleIT,DC=formation,DC=local" `
                         -ProtectedFromAccidentalDeletion $true

# Lister les OU protégées
Get-ADOrganizationalUnit -Filter {ProtectedFromAccidentalDeletion -eq $true} | Select-Object Name
Toujours protéger les OU importantes

-ProtectedFromAccidentalDeletion $true empêche une suppression d'un clic. Sur une OU qui contient des dizaines de comptes, c'est un filet de sécurité indispensable. C'est d'ailleurs le comportement par défaut via l'interface graphique.

TP guidé 2 — structure de l'École IT

Objectif : créer toute la structure d'OU de l'École IT (80 personnes, 8 serveurs, 60 postes). Durée : 60 min.

On factorise avec une petite fonction qui crée et protège chaque OU :

Administrator: Windows PowerShell
$BaseDN = "DC=formation,DC=local"

function New-ProtectedOU {
    param($Name, $Path)
    New-ADOrganizationalUnit -Name $Name -Path $Path
    Set-ADOrganizationalUnit -Identity "OU=$Name,$Path" -ProtectedFromAccidentalDeletion $true
}

# Racine
New-ProtectedOU -Name "EcoleIT" -Path $BaseDN

# Utilisateurs
New-ProtectedOU -Name "Utilisateurs"   -Path "OU=EcoleIT,$BaseDN"
New-ProtectedOU -Name "Direction"      -Path "OU=Utilisateurs,OU=EcoleIT,$BaseDN"
New-ProtectedOU -Name "Administration" -Path "OU=Utilisateurs,OU=EcoleIT,$BaseDN"
New-ProtectedOU -Name "Formateurs"     -Path "OU=Utilisateurs,OU=EcoleIT,$BaseDN"
New-ProtectedOU -Name "Stagiaires"     -Path "OU=Utilisateurs,OU=EcoleIT,$BaseDN"

# Ordinateurs
New-ProtectedOU -Name "Ordinateurs"   -Path "OU=EcoleIT,$BaseDN"
New-ProtectedOU -Name "Serveurs"      -Path "OU=Ordinateurs,OU=EcoleIT,$BaseDN"
New-ProtectedOU -Name "PostesTravail" -Path "OU=Ordinateurs,OU=EcoleIT,$BaseDN"

Vérifier le résultat :

Administrator: Windows PowerShell
Get-ADOrganizationalUnit -Filter * -SearchBase "OU=EcoleIT,DC=formation,DC=local" |
    Select-Object Name, DistinguishedName
dsa.msc   # contrôle visuel dans Active Directory Users and Computers
Définir une fonction en PowerShell

function Nom { param(...) ... } crée une commande réutilisable. Ici, New-ProtectedOU évite de répéter deux lignes par OU : on décrit la logique une fois, on l'appelle autant que nécessaire. C'est exactement l'esprit « script » du Jour 1.

Les outils d'administration

Trois façons d'ouvrir le classeur

On peut consulter et modifier l'annuaire de trois manières : à la souris (ADUC), au scalpel pour les attributs cachés (ADSI Edit, à manier avec prudence), ou en série par script (PowerShell AD). Trois portes vers le même classeur, pour des usages différents.

ADUC — Active Directory Users and Computers

Lancement : dsa.msc. La console graphique du quotidien. Activez Affichage → Fonctionnalités avancées pour voir tous les onglets.

ActionProcédure
Créer un utilisateurClic droit dans l'OU → Nouveau → Utilisateur
Créer un groupeClic droit dans l'OU → Nouveau → Groupe
Réinitialiser un mot de passeClic droit sur l'utilisateur → Réinitialiser le mot de passe
RechercherClic droit sur le domaine → Rechercher (F3)

PowerShell AD — pour l'échelle

Administrator: Windows PowerShell
# Découvrir les commandes AD disponibles
Get-Command *AD* | Group-Object Noun | Sort-Object Count -Descending | Select-Object -First 10

# Lire tous les attributs d'un objet
Get-ADUser Administrateur -Properties *

# Modifier un attribut précis
Set-ADUser Administrateur -Department "Direction"

# Inventaires utiles
Get-ADUser -Filter * | Select-Object Name, Enabled
Get-ADGroup -Filter * | Select-Object Name

ADSI Edit — l'éditeur expert

Lancement : adsiedit.msc. Il montre les attributs de bas niveau invisibles dans ADUC, et permet d'explorer les partitions de l'annuaire.

Outil à risque

ADSI Edit modifie l'annuaire sans garde-fou : une mauvaise manipulation peut corrompre AD. On l'utilise pour observer et comprendre, ou ponctuellement sur instruction précise. Pour le travail courant, on reste sur ADUC ou PowerShell.

TP guidé 3 — maîtriser les outils

Objectif : explorer ADUC, PowerShell AD et ADSI Edit, puis produire un rapport. Durée : 90 min.

Partie A — ADUC

  1. Ouvrir dsa.msc, activer les fonctionnalités avancées.
  2. Créer un utilisateur test dans l'OU Formateurs.
  3. Créer un groupe de sécurité Formateurs_Math et y ajouter l'utilisateur.

Partie B — PowerShell AD

Administrator: Windows PowerShell
# Créer 3 groupes de promotion
"B1","B2","B3" | ForEach-Object {
    New-ADGroup -Name $_ -GroupScope Global -Path "OU=Stagiaires,OU=Utilisateurs,OU=EcoleIT,DC=formation,DC=local"
}

# Repérer les comptes désactivés et les groupes
Get-ADUser -Filter {Enabled -eq $false} | Select-Object Name
Get-ADGroup -Filter * | Select-Object Name

Partie C — Rapport AD

Administrator: Windows PowerShell
$Report = @()
$Domain = Get-ADDomain
$Report += "Domaine : "          + $Domain.DNSRoot
$Report += "Niveau fonctionnel : " + $Domain.DomainMode
$Report += "Utilisateurs : "     + (Get-ADUser -Filter *).Count
$Report += "Ordinateurs : "      + (Get-ADComputer -Filter *).Count
$Report += "Groupes : "          + (Get-ADGroup -Filter *).Count
$Report += "OU : "               + (Get-ADOrganizationalUnit -Filter *).Count
$Report += "Contrôleurs : "      + (Get-ADDomainController -Filter *).Count
$Report | ForEach-Object { Write-Host $_ -ForegroundColor Cyan }
Le tableau et le pipe ForEach

$Report = @() crée un tableau vide ; += y ajoute des lignes. | ForEach-Object { ... } exécute une action pour chaque élément. On retrouve l'idée de « recette répétée » du Jour 1, appliquée à une liste.

Réflexes & contrôle

Réflexes anti-erreur
  • Pas de DNS sain → pas de promotion réussie. Tester Resolve-DnsName formation.local d'abord.
  • Après promotion, DNS du DC = 127.0.0.1, sinon l'ouverture de session casse.
  • Le nom de la forêt = le nom du domaine racine.
  • Mot de passe DSRM ≠ mot de passe administrateur.
  • Chemin LDAP → se lit de droite à gauche.
