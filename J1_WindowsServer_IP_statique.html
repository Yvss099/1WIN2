Au programme aujourd'hui

  1. Comprendre les modes réseau des machines virtuelles (NAT, Réseau interne, Bridge, Host-Only).
  2. Installer un Windows Server 2022 Standard Desktop Experience.
  3. Donner une IP statique au serveur — en interface graphique et en PowerShell.
  4. Comprendre les profils réseau (Public / Privé / Domaine) et configurer le pare-feu.
  5. Administrer le serveur à distance depuis un poste client (RSAT + WinRM), sans RDP.

Prérequis — Une VM Windows Server 2022 (4 Go RAM, 60 Go disque, 2 vCPU), une VM cliente Windows 10/11, l'ISO du serveur, et l'hyperviseur VirtualBox. C'est le premier jour : aucune notion antérieure n'est supposée.

Le réseau des machines virtuelles

Avant de poser une IP, il faut savoir sur quel réseau la carte de la VM est branchée. Ce choix se fait dans l'hyperviseur, avant de toucher à Windows. C'est l'étape que tout le monde rate.

La métaphore de l'immeuble de bureaux

Imaginez vos VM comme des bureaux dans un immeuble. La carte réseau est la porte du bureau. Le mode réseau décide sur quoi cette porte donne :

  • Réseau interne = un couloir privé qui ne relie que les bureaux du même étage. Invisible depuis la rue. Parfait pour que le serveur et le client se parlent entre eux.
  • NAT = la porte de service avec un standard : vous pouvez appeler vers l'extérieur (Internet), mais personne de la rue ne peut entrer chez vous directement.
  • Bridge (Accès par pont) = brancher le bureau directement sur la prise murale de l'immeuble : la VM devient une vraie machine du réseau physique de la salle.
  • Host-Only = un interphone privé entre la VM et votre PC hôte uniquement.

Le piège : une seule carte ne suffit pas

Le cahier des charges demande deux choses contradictoires pour une seule carte :

  • une IP fixe 192.168.10.10 pour que le client retrouve toujours le serveur au même endroit ;
  • un accès Internet (tester 8.8.8.8, installer RSAT, telnet google.com).
Piège classique — la passerelle fantôme

En Réseau interne seul, l'adresse 192.168.10.1 indiquée comme passerelle n'existe pas : aucun équipement ne route vers Internet sur ce couloir privé. Résultat : Test-NetConnection 8.8.8.8 échoue et Resolve-DnsName google.com reste muet. Cette .1 ne deviendra réelle que plus tard, quand ce serveur jouera lui-même le rôle de passerelle / DNS. Sur le J1, elle est inactive.

La solution : deux cartes — NAT + Réseau interne

On donne deux portes à chaque VM. Chacune fait un seul métier :

À retenir

La carte NAT porte la route par défaut et le DNS Internet (fournis automatiquement). La carte interne ne porte que l'IP statique du lab — sans passerelle. Deux métiers, deux cartes, zéro conflit.

Configurer les deux cartes dans l'hyperviseur (GUI)

VirtualBox — VM éteinte : Configuration → Réseau.

  • Onglet « Carte 1 » : cocher Activer la carte réseau → Mode d'accès réseau = NAT.
  • Onglet « Carte 2 » : cocher Activer la carte réseau → Mode = Réseau interne → Nom = lab (le même nom sur le serveur et sur le client, sinon ils ne se voient pas).
Bonne pratique

Faites le même montage à deux cartes sur la VM cliente. Elle aura ainsi Internet (NAT) pour installer RSAT, et l'IP 192.168.10.50 sur le réseau interne pour atteindre le serveur.

Accès RDP depuis l'hôte (astuce labo)

Le réseau interne « lab » est cloisonné : seules les VM se voient entre elles, votre PC hôte n'y a pas accès. Pour ouvrir une session RDP depuis l'hôte vers le serveur (pratique en formation, sans dépendre du réseau de la salle), on ajoute une redirection de port sur la carte NAT — sans toucher au réseau interne. La même recette fonctionne à l'identique sur chaque poste de la promo.

  • VM éteinte : Configuration → Réseau → Carte 1 (NAT) → Avancé → Redirection de ports.
  • Ajouter une règle : Nom RDP, Protocole TCP, IP hôte 127.0.0.1, Port hôte 13389, IP invité 10.0.2.15, Port invité 3389.
Hôte (PowerShell / Terminal)
# Équivalent en ligne de commande, VM éteinte (adaptez le nom de la VM) :
VBoxManage modifyvm "SRV-FORM-XX" --natpf1 "RDP,tcp,127.0.0.1,13389,10.0.2.15,3389"

Depuis l'hôte, le client RDP se connecte alors sur localhost:13389. Prérequis sur le serveur : Bureau à distance activé et pare-feu RDP ouvert (voir §3.3).

Pourquoi 13389 et pas 3389 ?

Si l'hôte est lui-même un Windows, le port 3389 peut déjà être pris (RDP de l'hôte, plages réservées Hyper-V/WSL) — la redirection échoue alors silencieusement. Un port hôte décalé comme 13389 évite tout conflit.

Et l'IP statique alors ?

La redirection vise 10.0.2.15 : c'est l'adresse DHCP de la carte NAT (Carte 1), pas l'IP statique du lab. L'IP statique 192.168.10.10 reste sur la carte interne (Carte 2) pour le dialogue serveur ↔ client. Le service RDP écoute sur toutes les cartes à la fois, donc il répond aussi bien sur le réseau interne (depuis le client VM) que via la redirection NAT (depuis l'hôte). Ne posez jamais d'IP statique sur la carte NAT, sinon la redirection ne tombe plus sur personne.

Installer Windows Server 2022

Vous êtes prestataire IT. Le client veut une machine clean, en réseau, documentée. On installe l'édition avec interface graphique, on partitionne, on définit le mot de passe.

Édition : Desktop Experience, pas Core

Au choix de l'édition, sélectionnez exactement Windows Server 2022 Standard (Desktop Experience).

Piège classique

La ligne « Windows Server 2022 Standard » tout court, c'est Server Core : aucune interface graphique, que de la ligne de commande. Pour ce premier jour, ce n'est pas ce qu'on veut. Vérifiez bien la mention (Desktop Experience).

Partitionnement & premier démarrage

  1. Langue / format / clavier : Français.
  2. Type d'installation : Personnalisée (pas « Mise à niveau »).
  3. Sélectionner le disque → Nouveau → taille proposée → Appliquer. Windows crée 3 partitions (système, MSR, Windows) : c'est normal. Sélectionner la grande partition → Suivant.
  4. Au premier démarrage, mot de passe Administrator : P@ssw0rd123!

Renommage, fuseau horaire et RDP se font ensuite via Gestionnaire de serveur → Serveur local (voir §3.3).

IP statique — version GUI

La métaphore de la boîte aux lettres

Une IP dynamique (DHCP), c'est une chambre d'hôtel : on vous en réattribue une au hasard à chaque séjour. Pratique pour un client de passage. Mais un serveur, c'est une maison : il lui faut une boîte aux lettres fixe, gravée sur la porte, pour que le facteur (le client) sache toujours où déposer le courrier. C'est tout le sens de l'IP statique.

On configure uniquement la carte interne en statique. La carte NAT, elle, reste en automatique (DHCP) : c'est elle qui fournit Internet, on n'y touche pas.

Poser l'IP statique sur la carte interne

  1. Gestionnaire de serveur → Serveur local → cliquer sur le nom de la connexion Ethernet qui correspond au réseau interne.
  2. Clic droit sur la carte → Propriétés.
  3. Double-cliquer sur Protocole Internet version 4 (TCP/IPv4).
  4. Cocher Utiliser l'adresse IP suivante :
    • Adresse IP : 192.168.10.10
    • Masque de sous-réseau : 255.255.255.0
    • Passerelle par défaut : laisser vide sur le J1 (voir l'encadré).
  5. Serveur DNS : laisser vide aussi (la résolution Internet passe par la carte NAT).
  6. OK → OK.
Pourquoi passerelle & DNS vides sur la carte interne ?

Si vous mettez 192.168.10.1 en passerelle sur la carte interne, Windows crée une seconde route par défaut vers une adresse qui ne répond pas. Au mieux elle est inactive, au pire elle perturbe l'accès Internet fourni par la NAT. Une seule carte porte la route par défaut : la NAT. Le cahier des charges (passerelle/DNS en .1) ne prendra son sens qu'à partir du J2, quand ce serveur deviendra lui-même passerelle/DNS.

Renommer, fuseau horaire, Bureau à distance

Toujours depuis Gestionnaire de serveur → Serveur local :

  • Nom de l'ordinateur : cliquer sur le nom actuel (WIN-XXXXXX) → Modifier… → saisir SRV-FORM-XX (XX = vos initiales). Redémarrage requis.
  • Fuseau horaire : cliquer dessus → (UTC+01:00) Bruxelles, Copenhague, Madrid, Paris.
  • Bureau à distance : cliquer sur « Désactivé » → cocher Autoriser les connexions distantes → accepter le message du pare-feu.

Puis Menu Démarrer → Power → Redémarrer pour valider le renommage.

IP statique — version CLI (PowerShell)

En entreprise, on ne configure pas 50 serveurs à la souris. On écrit un script. Refaisons toute la configuration réseau en PowerShell, sans jamais toucher à l'interface. Ouvrez PowerShell en administrateur.

La métaphore poursuivie

La GUI, c'est aller graver la boîte aux lettres à la main, bureau par bureau. PowerShell, c'est dicter les instructions au serrurier : une fois écrites, il les applique partout, à l'identique, sans se tromper.

Les 6 commandes de diagnostic

Tapez chacune, observez la sortie, notez en une phrase ce qu'elle fait :

Administrator: Windows PowerShell
Get-NetAdapter              # liste les cartes physiques et leur état
Get-NetIPAddress            # les adresses IP posées sur chaque carte
Get-NetIPConfiguration      # vue d'ensemble : IP + passerelle + DNS par carte
Get-DnsClientServerAddress  # les serveurs DNS configurés
Get-NetRoute                # la table de routage (qui mène où)
Get-NetFirewallProfile      # l'état des 3 profils de pare-feu
Astuce

Si vous comprenez ces 6 commandes, vous tenez 80 % du diagnostic réseau Windows.

Repérer la bonne carte

Vous avez deux cartes. Il faut configurer la carte interne, pas la NAT. Pour les distinguer :

Administrator: Windows PowerShell
Get-NetIPConfiguration
# La carte NAT a déjà une IP automatique en 10.0.2.x (fournie par DHCP).
# La carte interne, elle, n'a pas d'IP utile (169.254.x.x) tant qu'on n'a rien posé.
# => c'est CELLE-LÀ qu'on va configurer.

On renomme les cartes pour ne plus se tromper (adaptez "Ethernet 2" au nom réel vu plus haut) :

Administrator: Windows PowerShell
# Carte interne -> "LAN" (celle qui portera l'IP statique)
Rename-NetAdapter -Name "Ethernet 2" -NewName "LAN"

# Carte NAT -> "NAT-Internet" (reste en DHCP, on n'y touche pas)
Rename-NetAdapter -Name "Ethernet" -NewName "NAT-Internet"
Le nom exact compte

Si votre carte ne s'appelle pas exactement Ethernet 2, la commande échoue avec « No matching … was found ». Recopiez le champ Name rendu par Get-NetAdapter, espaces compris.

Poser l'IP statique

Administrator: Windows PowerShell
# 1. Nettoyer une éventuelle ancienne config (évite "L'adresse existe déjà")
Remove-NetIPAddress -InterfaceAlias "LAN" -Confirm:$false -ErrorAction SilentlyContinue
Remove-NetRoute     -InterfaceAlias "LAN" -Confirm:$false -ErrorAction SilentlyContinue

# 2. Poser l'IP statique SUR LA CARTE INTERNE — sans passerelle (J1)
New-NetIPAddress -InterfaceAlias "LAN" `
                 -IPAddress 192.168.10.10 `
                 -PrefixLength 24
Le backtick ` en fin de ligne

L'accent grave ` dit à PowerShell « la commande continue à la ligne suivante ». Il sert seulement à aérer un appel long. Vous pouvez tout écrire sur une seule ligne sans backtick.

On a retiré -DefaultGateway volontairement

La carte NAT fournit déjà la route par défaut. Ajouter -DefaultGateway 192.168.10.1 sur la carte interne créerait une route vers une passerelle inexistante. On l'omet sur le J1.

DNS (optionnel sur le J1)

La résolution Internet passe par la carte NAT, donc rien à configurer sur la carte interne aujourd'hui. Pour mémoire, la commande qui le ferait :

Administrator: Windows PowerShell
# Inutile sur le J1 — la carte NAT fournit déjà un DNS Internet.
# À partir du J2, le serveur pointera vers lui-même :
Set-DnsClientServerAddress -InterfaceAlias "LAN" `
                           -ServerAddresses 192.168.10.10

Vérification finale :

Administrator: Windows PowerShell
Get-NetIPConfiguration

Profil réseau & pare-feu

La métaphore des trois tenues

Windows adapte sa sécurité selon « l'endroit » où il se croit, comme on s'habille selon le lieu :

  • Public = dans la rue, méfiant : portes verrouillées, on n'ouvre à personne (admin distante bloquée).
  • Privé = à la maison, en confiance : on accepte les voisins connus (le client peut administrer).
  • Domaine = au bureau, sous l'autorité du domaine Active Directory (pour plus tard).

Forcer le profil en Privé

Administrator: Windows PowerShell
Get-NetConnectionProfile     # quel profil est actif sur chaque carte ?

# Forcer la carte interne en Private (sinon l'admin distante échouera)
Set-NetConnectionProfile -InterfaceAlias "LAN" -NetworkCategory Private
Le piège n°1 du TP suivant

Si la carte interne reste en Public, le WinRM (admin distante) sera bloqué et rien ne fonctionnera à la §6. C'est l'erreur la plus fréquente de la journée. Vérifiez-le maintenant.

Activer quelques règles de pare-feu

Administrator: Windows PowerShell
# Vérifier que les 3 profils sont actifs (Enabled = True)
Get-NetFirewallProfile | Select-Object Name, Enabled

# Autoriser le ping entrant (ICMPv4)
Enable-NetFirewallRule -DisplayName "File and Printer Sharing (Echo Request - ICMPv4-In)"

# Autoriser le groupe WinRM (admin distante, §6)
Enable-NetFirewallRule -DisplayGroup "Windows Remote Management"

# Créer une règle perso : HTTPS (443) en entrée
New-NetFirewallRule -DisplayName "Allow HTTPS" `
                    -Direction Inbound `
                    -Protocol TCP `
                    -LocalPort 443 `
                    -Action Allow
Bonne pratique

Écrivez toute cette séquence dans un fichier config_reseau.ps1 commenté. Demain, vous apprendrez à exécuter un script .ps1 d'un coup.

Administration distante (RSAT + WinRM)

En entreprise, on n'ouvre jamais de session RDP directe sur un serveur de prod : question de sécurité, de traçabilité, de productivité. On administre depuis son poste, via RSAT et WinRM. (Dans ce TP, le serveur est en 192.168.10.10 et le client en 192.168.10.50 — toujours sur le réseau interne.)

Côté serveur : activer l'écoute distante

Administrator: Windows PowerShell
Enable-PSRemoting -Force      # active WinRM, ouvre le port 5985, configure les services
Get-NetTCPConnection -LocalPort 5985   # doit afficher une ligne en "Listen"

Côté client : RSAT + liste de confiance

Administrator: Windows PowerShell
# Installer les outils d'administration distante (10-15 min)
Get-WindowsCapability -Online -Name "RSAT*" | Add-WindowsCapability -Online

# Hors domaine, déclarer le serveur comme "de confiance"
Set-Item WSMan:\localhost\Client\TrustedHosts -Value "192.168.10.10" -Force

Enter-PSSession vs Invoke-Command

Administrator: Windows PowerShell
# Tester le canal WinRM (doit répondre du XML "Microsoft Corporation")
Test-WSMan -ComputerName 192.168.10.10

# SESSION INTERACTIVE : on "entre" sur le serveur, on tape plusieurs commandes
Enter-PSSession -ComputerName 192.168.10.10 -Credential (Get-Credential)

# EXÉCUTION "ONE-SHOT" : une seule commande à distance, sans ouvrir de session
Invoke-Command -ComputerName 192.168.10.10 `
               -Credential (Get-Credential) `
               -ScriptBlock { Get-Service | Where-Object Status -eq "Running" }
La différence à retenir

Enter-PSSession = vous entrez dans le serveur pour une session interactive (plusieurs commandes d'affilée). Invoke-Command = vous envoyez une commande, elle s'exécute, vous récupérez le résultat — idéal pour une action unique, ou sur plusieurs serveurs en parallèle.

Captures : sortie réussie de Test-WSMan, prompt [192.168.10.10]:, et le serveur en vert dans le Gestionnaire de serveur.

Vérifications finales

Sur le serveur, après config :

Administrator: Windows PowerShell
hostname                          # doit afficher SRV-FORM-XX
Get-NetIPConfiguration            # IP statique présente sur la carte interne
Test-NetConnection 8.8.8.8         # Internet via la NAT -> PingSucceeded : True
Resolve-DnsName google.com         # résolution DNS -> retourne une IP
Le test qui prouve que le montage est bon

Test-NetConnection 8.8.8.8 réussit grâce à la carte NAT, pas grâce à la passerelle .1. Si ce test échoue alors que l'IP statique est bien posée, c'est presque toujours que la carte NAT manque ou est mal configurée — pas l'IP statique.

Réflexes & questions de contrôle

Les réflexes anti-erreur de la journée
  • Internet qui ne marche pas → vérifier la carte NAT, pas l'IP statique.
  • Admin distante qui échoue → la carte interne est-elle en Private ?
  • « L'adresse existe déjà » → Remove-NetIPAddress avant New-NetIPAddress.
  • Commande qui ne trouve pas la carte → recopier le Name exact de Get-NetAdapter.
  • Masque /24 ⇔ 255.255.255.0 ⇔ -PrefixLength 24 : trois écritures, une seule réalité.
